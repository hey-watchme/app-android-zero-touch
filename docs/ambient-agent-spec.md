# ZeroTouch 企画書

## 1. この文書の目的

この文書は、`ZeroTouch` の企画の軸と、完全実装までの筋道を簡潔に定義するためのものです。

このプロダクトは、Android タブレットを現場に置くだけで、会話や周辺の業務音声を
メモ、ナレッジ、エージェント行動へと段階的に変えていくことを目指します。

今後のモデル選定、データベース設計、バックエンド連携は `WatchMe` プロジェクトの
インフラを使う前提で進めます。モデルや DB の詳細は後続で確定します。

## 2. プロダクト名

暫定の製品名は `ZeroTouch` です。

この名前を当面維持する理由は以下です。

- 置くだけ、入力不要、教育不要という体験価値が名前に直接乗る
- Phase 1 の「メモ自動化」から Phase 3 の「エージェント化」まで一貫して使える
- ベタで説明しやすく、現場にも投資家にも伝わりやすい

将来的にサブタイトルを足す場合は以下のような形が考えられます。

- `ZeroTouch Memo`
- `ZeroTouch Knowledge`
- `ZeroTouch Agent`

ただし、現時点ではブランドを分けずに `ZeroTouch` 一本で進めるのが分かりやすいです。

## 3. コンセプト

### 一言でいうと

置くだけで、会話が業務とナレッジになる。

### 目指す世界

現場では、電話、口頭指示、予約、申し送り、確認、復唱といった会話の中に、
業務の重要情報が埋もれています。

`ZeroTouch` は、その会話を継続的に取り込み、まずはメモとして残し、次に
ナレッジとして蓄積し、最終的には自ら仕事を進めるエージェントに進化します。

## 4. 背景と課題

多くの現場業務は、依然として会話に強く依存しています。

- 記録は人手のメモや転記に頼っている
- 重要な情報がその場で消えていく
- 暗黙知がベテランや店長に閉じている
- 教育コストが高く、再現性が低い
- 非構造データである会話が活用されていない

つまり、会話が業務の入口であるにもかかわらず、データ化されていないことが
現場の生産性と再現性を下げています。

## 5. ソリューション

`ZeroTouch` は、タブレットを現場に置くだけで使えるゼロタッチ型プロダクトです。

- タブレットを設置する
- マイクで会話を取得する
- 音声を構造化可能な形に変換する
- メモ、カード、ナレッジ、実行アクションへつなげる

重要なのは、最初から完全自動を目指して無理をしないことです。

このプロダクトは、時間軸に沿って価値提供を積み上げます。

## 6. 提供価値のタイムライン

### Phase 1: メモ自動化

会話が自動でメモや付箋になる段階です。

想定価値:

- 電話内容がそのまま残る
- 予約、連絡、依頼、タスクの種がカード化される
- 転記作業が減る
- その日から便利だと感じられる

最初のユーザー体験はここです。

まずは、録音した音声が文字起こしされ、メモのように見返せることを成立させます。

### Phase 2: ナレッジ化

会話の蓄積から、暗黙知がマニュアル化されていく段階です。

想定価値:

- 注意事項や店舗ルールが蓄積される
- やり方や判断基準が自然言語で検索できる
- AI が「優しい先輩」のように振る舞う
- 教育コストが下がる
- 業務の再現性が上がる

ここで重要になるのは、単発の transcript ではなく、現場固有の文脈が
継続的にたまっていくことです。

### Phase 3: エージェント化

会話を理解し、判断し、プロアクティブに仕事を進める段階です。

想定価値:

- 予約登録
- スケジュール調整
- 顧客対応支援
- 確認リマインド
- 業務提案や次アクションの提示

ここは最も難易度が高い段階です。

ただし、Phase 1 と Phase 2 で蓄積したデータと文脈があるからこそ、
エージェント化の精度と実用性が上がります。

## 7. プロダクトの特徴

### 7.1 ゼロタッチ UX

- 設置するだけで価値が出る
- 操作を前提にしない
- 学習コストを最小化する

### 7.2 会話ベースの構造化

- 人間の復唱や確認会話を利用する
- 非構造な会話から事実を取り出す
- 入力フォームより自然なデータ取得を目指す

### 7.3 ローカル最適化

- 顧客名
- メニュー名
- 店舗ルール
- 社内用語
- 担当者の呼び方

こうした現場固有の文脈を蓄積することで、汎用 AI では代替しにくい
プロダクトになります。

### 7.4 置くほど価値が上がる

- データが蓄積する
- ナレッジが増える
- 精度が上がる
- 現場への適応が深くなる

この「使うほど手放せなくなる」性質が、長期的な競争優位になります。

## 8. ターゲット

初期ターゲットは、会話と復唱が業務の中心にある現場です。

- 放課後等デイサービス
- 飲食店
- 医療
- サロン
- 訪問サービス

共通条件は以下です。

- 電話や口頭確認が多い
- 暗黙知が多い
- 記録が人手に依存している
- 現場ごとの差分が大きい

## 9. ビジネス上の意味

このプロダクトの価値は、単なる音声文字起こしではありません。

- 会話履歴が残る
- 業務ナレッジが構造化される
- 現場固有の文脈が資産になる

この蓄積は、あとから簡単に代替できません。

`ZeroTouch` のモートは、店舗固有データ、会話履歴、運用ルール、
そしてそこから生まれる実務知識にあります。

## 10. 実装方針

### 10.1 端末

- Android ネイティブアプリとして実装する
- 想定端末は Xiaomi Redmi 系タブレット
- 最終的には専用機として常設利用する

### 10.2 利用するインフラ

- バックエンドは `WatchMe` プロジェクトのインフラを使う
- モデル、DB、API の詳細は後続で定義する
- そのため現時点では、特定のモデルや DB に縛られない設計を採る

### 10.3 許容する手段

実現手段は限定しません。

- オンデバイス処理
- クラウド STT
- クラウド LLM
- 外部ツール
- 人間によるレビュー
- ヒューマンインザループ

重要なのは、最終価値に対して合理的な手段を選ぶことです。

## 11. 機能パイプライン

このプロダクトの基本パイプラインは以下です。

1. 音声取得
2. STT
3. ファクト抽出
4. アノテーション
5. マッピング/整形
6. カード化
7. ナレッジ化
8. エージェント行動

各段階は差し替え可能であるべきです。

## 12. 最初の実装スコープ

最初は、メモのようなものから始めます。

具体的には以下です。

- タブレットで録音できる
- 録音した音声を文字起こしできる
- transcript を画面で見返せる
- 音声と transcript をひも付けて保持できる

この段階では、まだ高精度な抽出や自動実行は不要です。

最初の成功条件は明確です。

- 「置いて録ったら、まずメモになる」

この体験が成立して初めて、次のナレッジ化とエージェント化へ進めます。

## 13. ビジョン

`ZeroTouch` が最終的に目指すのは、入力のない世界です。

- 会話がそのままデータになる
- 業務が自然に構造化される
- AI が現場に常駐する

つまり、業務の入口そのものを押さえるプロダクトを目指します。
