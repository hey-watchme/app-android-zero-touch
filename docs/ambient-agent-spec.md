# Ambient Agent 企画兼仕様書

## 1. 文書の目的

この文書は、Android タブレット上で動く「アンビエントエージェント」アプリの
企画、最終像、最初の実装範囲を固定するためのベース仕様である。

今後プロジェクト構造を再設計する前提で、まず以下を明確にする。

- このアプリが最終的に何を実現するのか
- どの順番で実装フェーズを切るのか
- 最初のマイルストーンをどこに置くのか
- 端末、入出力、外部サービス、人間介在をどこまで許容するのか

## 2. 背景と狙い

現場には、会話、電話、口頭指示、予約対応、引き継ぎ、バイト指示、メモなど、
大量の非構造データが流れている。しかしそれらはその場で消えやすく、後から
検索・再利用・タスク化しにくい。

本企画の狙いは、その場の音声や入力を継続的に取り込み、現場で発生した事実を
構造化し、タスクカードやイベントカードとして保存・表示することである。

重要なのは、単なる録音アプリではなく、以下を実現することである。

- 現場の会話や音声から事実を拾う
- 事実を整理し、必要に応じて評価・補足する
- UI 上で扱えるカードへ変換する
- 後から検索、確認、修正、再利用できる
- 現場固有の文脈を蓄積して、精度を上げ続ける

## 3. プロダクトビジョン

### 3.1 一言でいうと

専用 Android タブレットを現場に置くだけで、現場の音声や操作入力から
「今やるべきこと」「起きたこと」「覚えておくべきこと」をカードとして
自動生成・蓄積するアンビエントエージェント。

### 3.2 最終的な姿

最終的には、タブレットが現場の知覚器官として振る舞う。

- マイクで音声を拾う
- 画面でカードを見せる
- タッチ、音声、簡易入力で人間が補正できる
- 必要に応じて外部 API、クラウド STT、LLM、ヒューマンインザループを使う
- 店舗や現場ごとの語彙、ルール、通称、業務フローを学習する

出力は単なる文字起こしではなく、例えば次のようなカードである。

- タスクカード: 19時予約の折り返し、仕込み追加、バイトへの指示
- イベントカード: クレーム発生、在庫切れ、電話予約、キャンセル
- ナレッジカード: 常連客の好み、業務上の注意、店舗固有ルール

## 4. 想定環境

### 4.1 端末

- 端末種別: Android タブレット
- 想定端末: Xiaomi Redmi 系タブレット
- 位置づけ: 専用機として常設するが、まずは通常のネイティブアプリとして成立させる

### 4.2 利用シーン

最初の具体例として、飲食店を主要ユースケースに置く。

例:

- 電話予約対応
- 席、時間、人数、名前の復唱
- バイトへの口頭指示
- 厨房やホールへの伝達
- 在庫や仕込みに関する判断
- クレームや例外対応
- 引き継ぎ事項の記録

### 4.3 使えるモーダル

このアプリは、タブレットで使えるモーダルを制約せず活用する。

- マイク入力
- 画面表示
- タッチ操作
- キーボード入力
- 音声入力
- 通知
- バイブレーションや音によるフィードバック
- ネットワーク経由の外部 API 呼び出し
- ヒューマンインザループによる確認・修正

## 5. 設計原則

- ゼロタッチ優先: まずは「置いておくだけで価値が出る」ことを目指す
- ただし完全自動に固執しない: 精度や運用を優先し、人手介在を許容する
- パイプラインを分離する: STT、抽出、注釈、出力整形を分離し、差し替え可能にする
- 現場文脈を蓄積する: 店舗固有名詞、略称、ルール、常連情報を後から学習可能にする
- 最初は小さく動かす: まずは録音済み音声を文字起こしできればよい

## 6. 機能スコープ

### 6.1 機能全体像

本アプリの処理は、以下の段階に分かれる。

1. 音声取得
2. STT
3. ファクト抽出
4. ファクトへのアノテーション追加
5. マッピング/整形
6. カード保存・表示

### 6.2 各段階の役割

#### Stage 1: 音声取得

端末マイクや既存録音機能から音声を取得する。

入力例:

- リアルタイム録音
- 手動録音
- 既存音声ファイル

出力:

- 音声ファイル
- 収録開始/終了時刻
- 端末情報
- 収録セッション ID

#### Stage 2: STT

音声ファイルから文字起こしテキストを生成する。

要件:

- まずは「録音したものを文字にできる」ことを最優先とする
- オンデバイスでもクラウドでもよい
- 実装は差し替え可能にする
- 将来的には話者区別、タイムスタンプ、部分結果も扱えるようにする

出力:

- transcript
- セグメントごとの開始/終了時刻
- 信頼度
- STT エンジン種別

#### Stage 3: ファクト抽出

文字起こしから、業務上意味のある事実を取り出す。

例:

- 予約日時
- 人数
- 顧客名
- 依頼内容
- 誰への指示か
- 期限
- 変更点
- 注意事項

ここでは LLM やルールベース抽出を使ってよい。

出力:

- fact の配列
- fact の種別
- 元テキストへの参照
- 抽出理由または根拠

#### Stage 4: アノテーション追加

抽出した fact を整理、評価、補完する。

例:

- 緊急度
- 重要度
- 未確定/要確認フラグ
- 店舗ルールとの整合性
- 類似過去イベントとの紐付け
- 人手確認の必要性

出力:

- annotation 付き fact
- confidence
- review_status

#### Stage 5: マッピング/整形

fact と annotation を、UI や外部連携で使える出力形式に変換する。

例:

- タスクカード
- イベントカード
- メモカード
- ナレッジカード

出力例:

- card_type
- title
- summary
- structured_fields
- action_items

#### Stage 6: 保存・表示

生成されたカードをアプリ内に保存し、一覧・詳細・確認導線を提供する。

将来的な保存先候補:

- ローカル DB
- Supabase
- 独自バックエンド

## 7. 最初のマイルストーン

### 7.1 目標

最初のマイルストーンは、録音した音声が文字起こしされ、アプリ上で確認できること。

ここでは、まだファクト抽出、注釈付け、カード最適化を完成させなくてよい。

### 7.2 範囲

必須:

- 端末で録音できる
- 録音ファイルを保持できる
- 録音ファイルを STT にかけられる
- 結果の transcript を画面で表示できる
- 失敗時に再試行できる

あると良い:

- 文字起こし履歴の一覧
- 処理中ステータス表示
- 音声ファイルと transcript の紐付け
- セグメント時刻の表示

やらないこと:

- 高精度なタスク抽出
- 現場文脈の自動学習
- 本格的なカード UI
- 複雑な権限設計
- マルチユーザー運用

## 8. フェーズ分割

### Phase 0: 土台

目的:

- 専用端末上で安定して録音・保存・再生できる

現状:

- すでに最小の録音/再生機能は存在する

### Phase 1: STT の成立

目的:

- 録音済み音声を文字起こしし、UI に表示する

完了条件:

- ユーザーが録音後に transcript を確認できる
- 失敗時の状態が分かる
- 音声とテキストがセッション単位で保存される

### Phase 2: 最小カード化

目的:

- transcript から最低限の fact を抽出し、簡易カード化する

例:

- 「19時 4名 田中」で予約カードを作る
- 「バイト A に発注を頼む」でタスクカードを作る

完了条件:

- 手動トリガーで fact 抽出できる
- カード一覧に最低限の title が出る

### Phase 3: アノテーションと確認導線

目的:

- 抽出精度が不十分でも、運用可能な確認フローを入れる

例:

- 要確認タグ
- 修正 UI
- 承認/却下

### Phase 4: 現場文脈の蓄積

目的:

- 店舗固有名詞、客名、よくある予約形式、略語などを学習対象として蓄積する

### Phase 5: アンビエント運用

目的:

- 手動録音中心から、より常時的・半自動的な知覚へ拡張する

例:

- 常時待機
- 発話区間検出
- 自動セッション化
- 通知や確認依頼

## 9. 初期ユースケース

### 9.1 電話予約

入力例:

- 「19時で4名ですね、田中様ですね」

まず必要なのは transcript であり、その後に以下へつなげる。

- 日時
- 人数
- 名前
- 予約イベントカード

### 9.2 口頭指示

入力例:

- 「18時までにドリンクの在庫見ておいて」

将来的な出力:

- タスクカード
- 担当候補
- 期限

### 9.3 引き継ぎ

入力例:

- 「冷蔵庫の左上に明日の仕込み分を入れてあります」

将来的な出力:

- ナレッジカード
- 注意事項カード

## 10. システム方針

### 10.1 アーキテクチャ方針

実装は段階ごとに差し替えできるようにする。

推奨する抽象境界:

- Audio Capture
- Transcription
- Fact Extraction
- Annotation
- Card Mapping
- Storage
- UI

各段階は、最初はダミー実装でもよい。

### 10.2 外部依存の扱い

以下は制限しない。

- クラウド STT
- クラウド LLM
- 外部ワークフロー
- 人間によるレビュー
- 手動修正 UI

重要なのは、各処理を固定実装にせず、交換可能なインターフェースで囲うこと。

## 12. 非機能要件

- 専用機として常時使えること
- 通信失敗時でもローカルで最低限成立すること
- エラー状態が UI 上で分かること
- 後から手で修正できること
- ログとデバッグ情報を一定程度残せること

## 13. リスクと論点

### 13.1 技術リスク

- 店舗雑音下での STT 精度
- 日本語固有名詞の誤認識
- 長時間録音のバッテリー消費
- 常時利用時の発熱
- ネットワーク依存の遅延や失敗

### 13.2 運用リスク

- 現場で本当に置きっぱなし運用できるか
- 録音対象の範囲や同意の扱い
- オペレーション負荷が増えないか
- 人手修正の負担が大きくなりすぎないか

## 14. 直近の実装方針

次に着手する実装は Phase 1 である。

具体的には以下。

1. 既存の録音機能をセッション単位で扱えるようにする
2. STT レイヤーをインターフェースとして導入する
3. 最初の STT 実装を 1 つ選んで接続する
4. transcript を保存する
5. transcript を画面上に表示する

この時点では、fact 抽出はまだダミーまたは未実装でよい。

## 15. 成功条件

この企画の第一段階の成功条件は、ユーザーがこう感じることである。

- 「置いて録音したら、まず文字になる」
- 「後からその内容を見返せる」
- 「この先、タスクカード化できそうだと分かる」

この感触が取れれば、次の段階として抽出、注釈、カード化に進める。
